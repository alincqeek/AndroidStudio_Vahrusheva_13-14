### Вопросы

1)Это класс который хранит данные экрана. Главная проблема - потеря данных при пересоздании экрана. Андроид уничтожает ACtivity и создает ее заново

2)MutableStateFlow - можно изменять значение. StateFlow - только для чтения. Создаем _uiState - мозжем менять и uiState - только даем чтобы ничего не менят. backing property pattern - есть приватное изменяемое поле и публичное читательское поле, контроллируем кто может менять данные

3)StateFlow живёт дольше Activity, есть утечки памяти.

4)StateFlow(логика экрана) - хранится в StateFlow, переживает повороты. remember(лучше для временного состояния)- внутри compose, при повороте исчезает. rememberSaveable(лучше для поля ввода текста) - сохраняется при повороне но только для простых задач 

5)Это класс для хранения данных. Генерируются -toString,equals,hashCode,copy. Copy похволяет изменять только одно поле

6)val uiState by viewModel.uiState.collectAsState() - тут мы подписываемся на Stateflow и следим за измнениями, при изменениях перерисовываем экран. Обновляется автоматически потомму что compose пересобирает только нужные части

7)Для лучшей логики, архитектуры, масштрабирования. Это поддерживает MVVM, инкапсуляцию, чистоту архитектуры

8)ViewModel уничтожается и все данные теряются, Ничего не сохраняется.DB для долгого хранения
